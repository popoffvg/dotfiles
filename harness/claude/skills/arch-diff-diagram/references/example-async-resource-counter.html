Async Resource Counter

Reference counting moves from an in-place counter serialized on one hot key to an append-only log reduced by GC in discrete slots. What changed, what's gone, what's new, what held — and why.

0The change in five slots

1Write path & contention — the headline

A "god object" (a Tengo template almost every workflow holds) makes every holder read-modify-write one counter key. The change replaces that shared write with an independent append.

A-0005Counter changes — increments and decrements — are appended to the system event log instead of written in place. Both directions matter: a god object takes a flood of increments, so logging only decrements would leave the conflict on the increment side.

2Source of truth & who reads it

The authoritative counter disappears. The log becomes the sole source of truth; the count is a materialized view that only GC computes.

A-0010No authoritative counters: the count is a materialized view of the log; any persisted value is a cache, always rebuildable by replay. Durability rests on the log (persisted in TxInfo, replayed on restart).

3Time & deletion

Truth stops being continuous. The count is defined only at a resource's GC slot boundary; deletion becomes eventual.

A-0009 · A-0012The GC pass is one discrete tick: invariants hold at slot boundaries on net totals, not continuously. Order within a slot is irrelevant; a transient mid-slot zero is never observed.

4Synchronization & locking

Going append-only removes the old sync point (the counter record's optimistic concurrency). It is replaced by a two-level lock: a global lock guards a per-resource lock registry; GC freezes one resource at a time.

A-0007 · A-0008Append-only has no sync point; the per-resource RW lock reintroduces one. Ordinary touch takes it shared; GC takes it exclusive for that resource's slot. A true global freeze happens only at the rare whole-map snapshot / compaction.

5Signatures as diffs

The behavioral contract shifts more than the shapes. No Go signatures are frozen yet (the async-gc branch has no code — evidence: overview §4), so these are behavior diffs, flagged as such.

6What did not change

Guard against over-reading the diff. These anchors hold.

  • GC still owns deletion — it remains the single authority that marks a resource deleted; only when moved (slot boundary, not commit time).
  • The directed per-edge counter ResourceRefCounterByResource{source,target} is untouched — it already spreads writes; only the single total key was the hot spot.
  • The system-event engine is reused, not rebuilt — RemoveResourceRef is already an async event in its own transaction; the change makes every counter change a logged fact it consumes.
  • "Zero holders ⇒ deleted ⇒ excluded from most operations" still holds — evaluated on net totals at slot boundaries instead of continuously.
