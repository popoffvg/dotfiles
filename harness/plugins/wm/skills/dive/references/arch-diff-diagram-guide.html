Information design for hard-systems review

The Architecture DIFF Diagram

One picture, two panels — current beside planned — that lets a reviewer answer what changed, what's gone, what's new, what held, and why in under 30 seconds. Rules first, each proven on one running example: making a resource reference-counter asynchronous.

1Goal & audience

The diagram exists to be reviewed, not admired. Its single job: collapse the reviewer's reading time. Design every choice against that.

  • Serve the reviewer, not the author. The author already knows the change — the diagram is a transfer, not a diary.
  • A reader must answer five questions from a glance: what changed · what's removed · what's new · what stayed · the one load-bearing "why". If any is unanswerable, the diagram failed.
  • 30-second budget. Headline legible standing back from the screen; detail rewards a second read (§9).
  • Show the seam, not the system. Diagram only what the change touches plus its immediate anchors.

2Layout & alignment discipline

The diff is done by the reader's eye, so hold the geometry still. Unchanged anchors keep the same position in both panels; only changed things move or recolor.

  • Two panels, shared coordinate grid. The counter-write actor sits at the same x,y in CURRENT and PLANNED. Movement then means change — it isn't noise.
  • Side-by-side when panels are shorter than wide (dependency graphs); stacked when tall (pipelines, timelines) so each panel keeps full width.
  • Wide/tall content scrolls inside its own box (overflow-x:auto) — the page never scrolls sideways.
  • Anchor with a spine: shared actors on the left, shared store on the right, in both panels.

3Visual encoding & the one legend

Four states, one palette, never color alone. Each state carries a glyph and a word so it survives grayscale printing and colorblind readers.

  • Fixed semantics: added=green+, removed=red✕ (or strike), changed=amber~, unchanged=grey (recede). Same everywhere, forever.
  • Redundant encoding. Hue + glyph + label. Deuteranopia can't split red/green — the +/✕ does.
  • Unchanged must recede: muted grey, thin stroke. It's context, not content.
  • One legend, always visible, near the panels. A diagram whose colors need memorizing is a quiz.

4Edge & arrow language

Arrows carry meaning. If control flow and data flow share one arrow style, the reader guesses. Give each relation a distinct line.

  • Reserve a stroke per relation: control solid → · data thick/double ⇒ · dependency dotted ⋯▷ · derive/refresh curved ⟳. Never overload one.
  • Append-only log = a lane of stacked cells growing one direction; new records append at the head, nothing rewrites earlier cells.
  • Materialized view = a derive arrow (⟳) from log → value, labeled with the reducer (here: net = Σ incr − Σ decr). The value is downstream, never a source.
  • Only GC reads the log/value here — draw that as the single consumer; don't imply holders read it.

5Representing time

The sync world has a continuous truth ("zero holders ⇒ deleted, instantly"). The async world defines truth only at slot boundaries. Don't draw a discrete state as if it were continuous — that lies.

  • Continuous state = a solid level line; every instant is defined.
  • Discrete/slot state = sampled dots at slot edges with a dashed hold between — the value between slots is explicitly undefined, not "still true".
  • Eventual consistency = a shaded delay window from "became net-zero" to "observed at next slot". Label the window; don't pretend the transition is instantaneous.
  • Mark the trigger for the next slot (timer ∨ log-size threshold) on the axis.

6Representing locks

The hardest thing to draw. A lock has two independent dimensions — scope (what it covers) and mode (how it's held). Encode them separately, then "stop-the-world" reads off the picture.

  • Scope = the box you draw the lock band around. Per-resource band hugs one resource; global band wraps the whole map.
  • Mode = fill. Shared/read = hatched, porous (many can enter). Exclusive/write = solid, opaque (one holder, everyone else waits).
  • Stop-the-world is scope × exclusive. Per-resource exclusive freezes one resource while the rest stay concurrent. Global exclusive freezes everything — reserve for the rare whole-map snapshot/compaction.
  • Show the two-level acquire order: global lock (short) guards the registry → then the per-resource lock. Draw it as nesting, not two peers.

7Concurrent shared state & contention

The whole point of the change is removing a hot spot. Draw the contention so its disappearance is the headline.

  • Show the fan-in. Many actors → one key = many converging arrows on one node. Contention is visually the pile-up.
  • Mark the hot node: a warning ring / "conflict" badge, plus the failure mode (ErrConflict, retry storm, sweeper panic @5000).
  • In PLANNED, arrows fan out to independent appends — no shared write target. The absence of convergence is the fix; let the whitespace speak.
  • Keep both panels' actor set identical and positioned identically (§2) so only the target topology differs.

8Signatures-as-diffs

Put the contract change in code, right beside the arch panel it explains — a red/green line diff. When behavior changes but the signature doesn't, say so explicitly; silence reads as "unchanged".

  • Real -/+ diff lines, same green/red semantics as the diagram.
  • Anchor each diff to its panel node (same label) so contract ↔ topology is one glance.
  • No signature change ≠ no change. Flag behavioral-only shifts with an explicit note.

9Annotations & rationale

Tie each change to its why — but sparingly, and layered. Headline reads first; rationale rewards the second read. Reference a decision id so the diagram stays a pointer, not a spec.

  • Progressive disclosure. The panel shows what; a small numbered callout gives why; the decision id (D-07) points to the full record elsewhere.
  • Cap it: annotate the load-bearing changes only. Every node labeled = nothing emphasized.
  • Attach the callout to the node, not floating in a corner. Proximity is meaning.

10Clutter control & anti-patterns

  • One idea per region. Contention in one panel-pair, locking in another, time in a third. Don't stack all three on one canvas.
  • Omit the unchanging majority. Draw the seam and its anchors; a full system map buries the diff.
  • No decoration. No 3D, drop shadows, gradients, skeuomorphism. They add ink, not meaning.
  • No legend-less color. Covered in §3 — it's the most common failure, so it's also an anti-pattern.
  • No arrow spaghetti. If lines cross more than they inform, split into two smaller diagrams.

11Checklists

Reviewer — the diagram is done when I can, in one pass:

Answer all five slots (changed/removed/new/same/why) without asking the author.
Name the removed hot key and its failure mode.
See that unchanged anchors sit in identical positions across panels.
Read every color's meaning from the on-page legend.
Tell control from data from derive from dependency edges.
See when the count exists (slot boundaries) vs when it's undefined.
Distinguish per-resource freeze from global freeze at a glance.
Trace each headline change to a decision id.

Author — before I ship it:

Every element falls into exactly one of the five slots — else cut it.
No color without a paired glyph/label; grayscale still reads.
Unchanged nodes are muted and pinned in place.
One idea per region; no diagram carries three concepts at once.
Discrete-time state is dashed/sampled, never drawn continuous.
Lock scope = box, mode = fill; stop-the-world reads off the picture.
Signature diffs beside their nodes; behavior-only changes flagged.
Zero decoration (no 3D/gradient/shadow); legend present.

12Why HTML, and how

  • Diffable & reviewable in the PR — the diagram lives beside the code it describes; git shows what changed in the picture too.
  • Searchable — labels are text, not pixels. Ctrl-F "ErrConflict" lands on the node.
  • Theme-able & no tool lock-in — prefers-color-scheme gives light/dark free; no binary editor, no license, no export step that drifts from source.
  • Accessible — real text, role="img" + aria-label on every SVG, semantic sections.
Minimal implementation. One file, CSS in a <style>, no external fetch. CSS custom properties hold the diff palette (swap once, recolors everything). CSS grid for the two panels; inline <svg> for graphs (use marker + pattern for arrowheads and lock hatching — as this page does). @media (prefers-color-scheme:dark) plus :root[data-theme] overrides so a host theme toggle wins both directions. Wide SVGs go in an overflow-x:auto box so the page body never scrolls sideways.

This guide is itself the demonstration: view source, diff it, print it, flip your OS theme.
